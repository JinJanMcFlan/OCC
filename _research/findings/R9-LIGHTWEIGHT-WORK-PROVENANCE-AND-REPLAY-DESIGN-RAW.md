A5 — Work Provenance and Replay Audit [OCC]The integration of autonomous cognitive capabilities within deterministic orchestration frameworks demands a rigorous approach to work provenance, state tracking, and execution replay. Within the Overseer Command Center (OCC) architecture, the Router acts as the exclusive mechanism for bounded work, capability assignment, scheduling, and outcome recording, while the Research Engine manages hypotheses, contradictions, and decision-ready handoffs. To support these functions truthfully—without introducing external database dependencies, event-sourcing frameworks, or graph databases—OCC relies on a purely JSON/in-memory data posture. This strict architectural constraint necessitates a lightweight yet robust provenance model capable of capturing task-fact evolution, routing rationales, capability snapshots, and execution outcomes, while simultaneously supporting deterministic time-travel debugging.The analytical imperative of this report is to evaluate existing provenance and replay patterns, identify critical failure modes inherent to in-memory architectures, formulate an OCC-specific improvement hypothesis, and define the minimum first-build requirements and B1-ready test cases necessary to validate this boundary.Existing Provenance, Audit-Trail, Replay, and Version-History PatternsTo establish a foundation for OCC’s JSON-based provenance and replay mechanisms, it is necessary to examine how adjacent operational systems, distributed workflows, and semantic web standards resolve the tension between state mutation and historical immutability. The challenge lies in tracking changes and lineage without relying on heavy persistence architectures.The W3C PROV Data Model and JSON SerializationsThe W3C PROV family of specifications provides a foundational, domain-agnostic conceptual data model for provenance. The standard defines provenance as information concerning entities, activities, and people involved in producing a piece of data or an object, enabling subsequent assessments regarding quality, reliability, or trustworthiness. The PROV model captures these relationships through three core components, which are highly applicable to OCC's routing and research logic.PROV ComponentDefinition within StandardApplication within OCC ArchitectureEntitiesPhysical, digital, or conceptual things in the world that exist at a point in time.Initial task facts, raw research documents, JSON output payloads, and ratified MainOrch decisions.ActivitiesDynamic processes that consume, process, or produce entities over a period of time.Routing decisions, model execution attempts, deterministic evaluations, and human approval gates.AgentsEntities responsible for an activity, representing individuals, organizations, or software.Specific AI models, external tools, deterministic rule sets, or human users interacting with the Base UX.The standard captures relationships such as derivation (linking an output to an execution attempt), usage (linking an execution attempt to input task facts), and attribution (linking an execution to a capability or model). For web and lightweight applications, this complex relational model is serialized via PROV-JSON or PROV-JSONLD. PROV-JSON leverages native data types and an indexed representation where assertions are grouped by type, allowing fast client-side lookups without requiring the traversal of an entire document. PROV-JSONLD adds semantic markup to integrate with Linked Data applications while maintaining a self-contained JSON structure suitable for in-memory processing.In distributed workflow management systems, such as the Common Workflow Language (CWL), the PROV standard is utilized to capture per-step provenance fragments during runtime. These fragments are subsequently collated into a contiguous record, demonstrating that deep, relational provenance can be maintained dynamically without requiring a centralized, monolithic tracking database. This distributed approach to provenance collection is directly transferable to OCC's need to track disconnected tasks, delegations, and research runs in memory.Durable Execution and State Replay PatternsIn modern distributed workflow engines—such as Temporal, Cadence, Restate, and the Azure Durable Task Framework—state is maintained not by saving continuous point-in-time snapshots of the application's memory heap, but by recording an append-only event history of non-deterministic operations. This approach relies on the principle of "Durable Execution." If the core workflow code is strictly deterministic, the engine can recover the exact current state after a crash, suspension, or split-gate wait state by re-executing the code from the beginning and feeding it the historical results of past activities.When the workflow logic encounters an execution boundary, such as a scheduled task or an external API call, the engine consults the JSON-formatted execution history. If the task is marked as completed in the history, the engine suppresses the live execution, injects the recorded output back into the workflow as if it had just executed, and continues to the next step. This capability allows engineers to download a JSON workflow history from a production environment and replay it locally in a debugger.The primary limitation of this pattern is its strict enforcement of determinism. Any modification to the workflow's topology—such as adding a timer, changing an activity type, or altering execution order—results in a non-deterministic failure during replay because the recorded history no longer aligns with the code path. Furthermore, traditional durable execution systems require heavy backing datastores to manage the event histories and handle concurrent lineage tracking, which conflicts with OCC's purely in-memory mandate. Therefore, OCC must abstract the replay mechanism away from the underlying persistence layer.JSON Patch and Delta Tracking PatternsTracking the evolution of mutable state, such as OCC's task facts, presents a significant challenge in an in-memory system. Task facts update iteratively as ambiguities are resolved, human approvals are granted, or users alter parameters like urgency. Storing full, deep copies of a JSON object at every state change rapidly leads to memory bloat and garbage collection latency. These systems store only the initial JSON state and the sequential array of patches. This delta-tracking pattern enables the reconstruction of any historical state in-memory without maintaining redundant full-object copies, aligning perfectly with OCC's lightweight architectural constraints while preserving a rigorous audit trail of user and system intent.Deterministic Replay for Non-Deterministic AI AgentsDebugging AI orchestration introduces a paradigm failure for traditional software testing. AI model sampling, variable API latency, and evolving external tool environments are inherently non-deterministic. Every agent run represents a unique snowflake of time-dependent state. To achieve time-travel debugging in AI systems, a specialized record-and-replay pattern is utilized by frontier observability tools.During the initial execution, the orchestrator runs normally, but every external interaction encompassing LLM prompts, tool invocations, and network API calls is intercepted and logged as a structured JSON payload. This forms a complete chronological tape of non-deterministic inputs and outputs. During the replay phase, the live LLM client and external tools are substituted with deterministic stubs that yield the recorded JSON responses verbatim when queried. This strictly isolates the deterministic routing and orchestration logic from the non-deterministic cognitive execution. By feeding historical model outputs into new routing logic, engineers can verify if an update to the orchestrator's algorithms produces the exact same sequence of tool selections and logic branching, ensuring that adjustments behave precisely as intended prior to deployment.In-Memory Journaling and the Memento PatternTo maintain application state purely in memory without persisting to a database, the Memento software design pattern is frequently deployed. An Originator object produces snapshots, a Memento holds the captured state, and a Caretaker manages the chronological history. In modern implementations, maintaining unbounded history quickly leads to memory exhaustion. Best practices dictate using incremental delta mementos rather than full snapshots, enforcing strict upper limits on history length, and utilizing deep copies only when dealing with mutable nested collections to prevent referential contamination. This pattern underscores the necessity of separating the active working state from the historical audit trail within the OCC architecture to maintain runtime performance.Failure Modes in Provenance and Replay SystemsImplementing provenance and replay mechanics strictly within an in-memory JSON environment exposes the architecture to several critical failure modes. Analyzing these vulnerabilities is necessary to prevent structural compromises in OCC's implementation.Failure Mode 1: Rewritten History and Loss of Contextual TruthA common architectural flaw occurs when systems update a centralized capability registry or a task-fact record in place, obliterating the historical context of prior routing decisions. For example, if a routing decision was made based on a specific model's high confidence score and favorable privacy posture at a given time, but the Capability Registry is subsequently updated to reflect an outage or a degraded confidence score, a naive system will display the updated registry values when an administrator reviews the historical execution. This results in a disconnected, untruthful audit trail where the router's prior decision appears irrational or fundamentally flawed, destroying the explainability of the routing rationale. The broader implication is a collapse of user trust; provenance must capture the exact capability evidence snapshot that existed at the precise millisecond of the routing decision.Failure Mode 2: Missing Decision Context and Blind OmissionsMany API-first workflows log the final output or the terminal error message but systematically fail to capture the chain of custody and the rejected alternatives leading up to the result. In an AI orchestrator, capturing the output JSON payload is entirely insufficient for provenance. If a task requires human approval or fails a hard constraint during the pre-scoring eligibility filters, the system must explain the mechanistic reasoning behind the event. Failure to record the evaluation of OCC's five routing factors against the available capabilities means the system cannot answer basic diagnostic queries. Without capturing the rejected alternatives and the specific hard restrictions that triggered their rejection, the replay mechanism becomes an opaque black box that displays execution paths without illuminating the underlying deterministic logic.Failure Mode 3: False Immutability ClaimsSystems operating in-memory or utilizing simple JSON files on disk often implement content hashing to verify artifact integrity, subsequently asserting to users that the records are immutable. A cryptographic hash only provides mathematical verification of content at a specific point in time; it does not structurally prevent the underlying JSON file from being deleted, overwritten, or corrupted during a memory flush or an application restart. Overstating the security or immutability of an in-memory JSON ledger creates false confidence and misaligns with security auditing standards. OCC must accurately describe its provenance as a verifiable, append-only chronological tape relying on application-level governance, rather than masquerading as a cryptographically secure, append-only database.Failure Mode 4: Overbuilt Event Systems and Memory SaturationTo achieve strict provenance and state reconstruction, architects often reflexively introduce complex Event Sourcing frameworks and Command Query Responsibility Segregation architectures. Translating these enterprise-grade persistence architectures into a pure JSON/in-memory boundary leads to catastrophic memory bloat, high serialization overhead, and unmanageable garbage collection pauses within the JavaScript engine. Attempting to track every micro-mutation of the UI state or minor internal variable within a centralized event stream directly violates the lightweight, standalone imperative of OCC. The ripple effect of this failure is severe latency and application instability. Event systems must be strictly bounded to major, state-changing orchestration milestones rather than keystroke-level telemetry.Failure Mode 5: Unexplained Relationships and Orphaned EvidenceIn systems where normal work routing and internal research tasks share infrastructure, failing to formally bind dependencies results in orphaned artifacts. If the Research Engine produces an evidence finding that triggers a capability update, but the provenance model does not explicitly link the resulting MainOrch decision handoff back to the original experimental evidence, the graph view fractures. The failure to maintain relational integrity prevents the accurate rendering of queue states and renders the Presentation Mode graph fundamentally untruthful, as it must invent fake links after the fact to display a cohesive UI. Provenance must be recorded at creation time to ensure relational authenticity.Candidate OCC Improvement HypothesisTo fulfill the requirements of the completed router research audits without violating the JSON/in-memory constraints, this report formulates an overarching improvement hypothesis for OCC's provenance architecture.The hypothesis asserts that OCC can achieve verifiable task provenance, explainable routing, and deterministic time-travel debugging strictly within an in-memory JSON architecture by implementing a concept termed the Tripartite Provenance Envelope. This envelope eschews a centralized event-sourcing database and avoids adopting specific graph databases. Instead, it attaches three distinct chronological artifacts directly to the lifecycle of an OCC Task Object, maintaining contextual integrity as the task moves through the system.The first component of the envelope is the RFC 6902 Delta Tape. For mutable entities, such as user-defined Task Facts, only the initial baseline state is stored in full. Subsequent clarifications, dependency resolutions, and priority shifts are appended strictly as JSON Patch objects, maintaining a lightweight version history that clearly outlines the intent of the mutation.The second component involves Point-in-Time Rationale Snapshots. When the Router commits to an execution, it embeds a static, frozen copy of the winning capability's registry evidence and a summary of the hard-rule eliminations directly into the Execution Attempt record. This completely insulates the historical audit trail from future drift in the Capability Registry.The third component relies on Deterministic I/O Stubs. Every external network boundary, including LLM token generation and tool execution, writes its exact request and response to an append-only JSON array owned by the Execution Attempt. This allows the Router to be replayed natively offline by feeding it the chronological stubs, verifying deterministic logic without incurring API costs or suffering from non-deterministic model hallucinations.This proposed mechanism improves upon traditional durable execution engines, which require rigid, unchanging workflow code and heavy backend infrastructure. Furthermore, it improves upon standard LLM tracing tools that capture vast amounts of data but treat the orchestration logic as an opaque script. By strictly separating the deterministic orchestration logic from the non-deterministic cognitive outputs at the JSON array level, OCC allows developers to safely replay a task through modified routing code against historical model outputs. This ensures that adjustments to the five routing factors or hard-rule gates behave precisely as intended before they are deployed into production.Minimum First-Build Provenance and Replay RequirementsTo implement the Tripartite Provenance Envelope successfully in the B1 build, OCC must define the minimum required JSON structures for shared records. These are not finalized database schemas, but structural prerequisites that guarantee compatibility with the current JSON/in-memory architecture while satisfying the audited research requirements.Task Fact Evolution and Mutable StateTask Facts must not be treated as a permanently frozen object, nor should they be blindly overwritten as new information emerges. OCC must establish a logical boundary containing base facts and an ongoing ledger of mutations. The base facts capture the initial user goal, expected result, urgency, user presence, output quality tier, and recognized missing information at the moment of ingestion.To track changes seamlessly, OCC requires an append-only array of RFC 6902 JSON Patch objects. If a user clarifies an ambiguous instruction or upgrades a task to an urgent priority, an explicit patch indicating the replacement operation is appended to the ledger. The current, active state of the task is continually derived by applying the chronological patches to the base facts in memory. This ensures that the system can always roll back to understand what the router believed at any prior specific phase of the workflow.Route Rationale and Capability SnapshotsWhen the OCC Router transitions a task from the queue to active execution, it must generate a routing decision record that statically embeds the decision context, thereby preventing future capability registry updates from rewriting history.This record requires an array detailing which capabilities were eliminated prior to scoring due to hard restrictions, such as missing tool access or privacy violations. Following elimination, the record must preserve the weighted scoring context, capturing the calculated scores based on the five routing factors for the remaining eligible models. Crucially, the record must capture a deep-copied JSON object of the chosen capability’s registry entry at that exact execution millisecond. This snapshot must include its operational health, known limitations, and semantic confidence score, establishing an irrefutable record of why the router made its choice based on the evidence available at the time.Execution Attempts and Deterministic Replay LogsA single routed task may require multiple execution attempts due to provider rate limits, tool failures, or human-in-the-loop rejections. Therefore, each execution attempt must maintain its own independent replay tape.This tape acts as an ordered JSON array of all non-deterministic external interactions. Each entry within the tape records a sequential step identifier, the operation type, the exact input payload transmitted, and the exact output payload received. During a replay debugging session, the OCC Router is placed into a specific offline mode. When the router attempts an external call, the engine intercepts the execution, matches the step identifier, and returns the payload directly from the replay tape. This mechanism is fundamental to proving that the OCC routing logic remains strictly deterministic.Approvals, Outputs, and Relational LinkageTo truthfully support the graph, queue, and Presentation Mode views, the outcome of an execution attempt must maintain referential integrity without relying on a dedicated graph database.Human approval gates must be recorded as discrete chronological events within the replay tape. If a user alters a parameter during the approval process, an RFC 6902 patch is generated against the execution state, maintaining the audit trail of human intervention. Furthermore, raw outputs, formal audits, accepted evidence, and proposed decisions must utilize lightweight PROV-JSON style relational pointers. A final artifact JSON must include a provenance metadata block containing a derivation key pointing to the execution attempt identifier, and an association key pointing to the specific agent or model utilized. This in-memory linkage allows the UI to construct directed acyclic graphs dynamically based on factual historical relationships.Outcome Feedback SegregationFollowing the completion or failure of an execution, OCC must safely record outcomes without automatically triggering volatile, system-wide semantic routing capability changes.The system requires a strict segregation of feedback. Direct operational telemetry facts—such as actual execution cost, latency, HTTP rate limit flags, and tool availability failures—are written to a dedicated operational log. This data can be utilized to temporarily suspend a capability via an automated circuit breaker without human intervention. Conversely, semantic feedback—encompassing user rejections, human reviews, or secondary model evaluations regarding the qualitative nature of the output—are logged as evaluation events. These qualitative assessments must require a formalized, human-governed review process before altering the Capability Registry's long-term semantic confidence scores, preventing the system from overreacting to a single anomalous output.B1-Ready Replay and Change-History Test CasesTo empirically prove that the Tripartite Provenance Envelope functions securely and reliably within the JSON/in-memory constraints of the Base UX and Router, the following B1 test cases must be successfully executed. The OCC improvement hypothesis must remain subject to ongoing scrutiny. The proposal would be disproved, forcing a substantial architectural redesign, if subsequent internal benchmarking reveals any of the following evidence.Catastrophic memory saturation poses the primary threat to the in-memory constraint. If an average Research Engine run generates thousands of sequential tool executions, the replay tape JSON arrays will grow significantly. Should heap-memory profiling demonstrate that OCC routinely exceeds typical desktop memory constraints solely due to storing deep-copied capability snapshots and expansive payload I/O, the append-only JSON approach is invalidated. This discovery would necessitate moving to an external database or aggressively truncating lineages, which subsequently breaks the promise of full reproducibility.Furthermore, deterministic replay relies entirely on exact structural matching. If the underlying JavaScript/TypeScript engine introduces unresolvable non-determinism in how it parses JSON keys from recorded stubs, or if rounding errors in the five-factor floating-point scoring logic result in cascading branching paths across different operating systems, the replay engine will constantly report false-positive divergences. This would render the time-travel debugging feature practically useless, as engineers could not trust the output of the divergence detector.Additionally, while JSON patches are highly efficient for storage, reconstructing a complex task state from a baseline object and hundreds of sequential patches requires iterative computational cycles. If benchmarking demonstrates that applying deep object patches introduces unacceptable latency that degrades UI responsiveness or router throughput, a traditional snapshot-heavy model must be substituted, forcing a compromise on memory footprint.Finally, the viability of the split-gate architecture remains an open question. If empirical observation reveals that post-selection commitment checks cannot accurately lock capabilities because external providers dynamically shift context limits, availability, or pricing between the scoring phase and the delayed execution phase, the frozen rationale snapshot will no longer reflect the reality of the execution. This would indicate that separating eligibility filtering from commitment verification is inherently unstable when interacting with highly volatile third-party AI providers.Beyond the Current GuardrailThe strict constraints of the current research explicitly bound OCC to JSON/in-memory data structures and deterministic routing logic. However, the intersection of AI orchestration, continuous evaluation, and state replay reveals several adjacent, frontier mechanisms that warrant tracking for future integration. These items do not alter the current B1 implementation scope or architecture, but are formally recorded here per the research operations protocol.The first frontier concept involves semantic circuit breakers via replay-driven pattern matching. Traditional operational circuit breakers trip predictably on HTTP 500 errors or recognizable rate limits. However, AI models frequently fail semantically, entering endless hallucination loops or repeatedly invoking tools with slightly mutated, incorrect parameters while maintaining a perfect HTTP 200 health status. This runaway semantic looping rapidly drains cost budgets. Adjacent security observability tools utilize high-throughput JSON log evaluation engines to detect temporal anomalies in real-time. By utilizing replay systems, engineers can train local classifiers against historical trace data to detect these behavioral loops. Should OCC implement a local embedding or classification model within the Base UX, it could theoretically analyze the in-memory replay tape in real time to sever execution loops autonomously, expanding safety beyond simple deterministic gates.The second concept addresses state verification for distributed agent swarms. Currently, OCC relies on simple chronological JSON arrays to track lineage. As multi-agent collaboration frameworks mature, multiple autonomous models operating asynchronously will need to agree on a shared context state without overwriting each other's task facts. Simple RFC 6902 arrays handle linear, single-user changes efficiently, but parallel swarm research will inevitably encounter merge conflicts. Content-addressed storage and Merkle Directed Acyclic Graphs (DAGs) are widely used in distributed version control and web3 infrastructures to resolve parallel state mutations seamlessly. This item is parked for now, as graph databases and content-addressed storage are explicitly excluded from the current build. However, it mandates a revisit trigger if OCC formally adopts parallel multi-agent execution frameworks that outgrow the capacity of linear JSON patching.The final concept involves generative trace exploration, or autonomous counterfactual replay. Currently, deterministic replay is envisioned as a static debugging tool utilized by human engineers to diagnose failure. However, the Research Engine could utilize this replay mechanic autonomously. If an execution fails to achieve its objective, the Research Engine could automatically fork the replay tape at the exact point of failure, subtly alter the prompt instructions or capability selection, and replay the execution in the background. Systems like LangGraph Time Travel already support checkpoint forking to explore alternative execution paths without re-running the successful, expensive prerequisite steps. This concept should be escalated when Main ResearchOrch begins formal architecture design for the automated hypothesis testing suite, as it possesses the potential to transform the replay engine from a passive audit tool into a core, active cognitive capability.
