Overseer Command Center (OCC) Architecture Research: General Router Task Facts and Evaluation MatrixFounder BriefThe central challenge of orchestrating multi-model artificial intelligence is that routing cannot simply consist of matching a natural language prompt to a generative execution model. When an architecture treats tasks uniformly—relying on a large language model to probabilistically determine the necessary actions upon receipt—the system inevitably becomes vulnerable to runaway compute costs, irreversible external errors, and catastrophic context collapse. The Overseer Command Center (OCC) is predicated on the requirement for a deterministic, capability-driven routing layer. To make repeatable, cost-effective, and safe routing decisions, this router must extract a structured, highly specific set of facts from every user request before any execution or delegation begins.This research investigates the structured facts OCC must identify to reliably understand, compare, split, schedule, approve, and route complex work across diverse task types. The investigation operates under the strict architectural constraint that OCC functions as a standalone, provider-neutral application acting as a stable control layer above swappable AI drivers. The routing engine must not be hardcoded to favor specific models or specific domains, but rather must evaluate requests based on universal computational and operational properties.The analysis reveals that AI routing is fundamentally an operations research, triage, and distributed resource-allocation problem. By examining adjacent domains—hospital emergency triage algorithms, high-performance computing schedulers, distributed microservice state machines, and military command structures—profound operational mechanisms emerge that translate directly to AI orchestration. For instance, the medical triage system separates a patient's immediate medical urgency from their predicted diagnostic resource consumption. Similarly, OCC must separate a task's time-sensitivity from the number of computational tools or files it requires. High-performance computing schedulers utilize backfill algorithms, proving that OCC should actively differentiate between large, synchronous blocking tasks and smaller, asynchronous tasks that can run in the background to maximize system utilization without delaying priority workloads.This report establishes that the OCC router must act as an advanced classifier that constructs a formalized "Task Fact Contract." This contract serves as a deterministic, provider-neutral ledger detailing the work's objective, urgency, risk profile, privacy constraints, and resource dependencies. It acts to transform an ambiguous, unstructured user prompt into a strict data object. By identifying whether work is urgent or deferrable, simple or multi-part, safe or high-risk, and synchronous or autonomous, OCC can apply mathematical routing rules based on cost, model fit, user presence, and output quality tiers without relying on the unpredictable nature of zero-shot model reasoning.The findings are distilled into five actionable architectural sections. First, a comprehensive definition of the necessary task facts is provided, mapping their exact definitions, their impact on routing decisions, and the fallback protocols required when these facts are unknown. Second, an evaluation matrix is constructed to categorize tasks by their fundamental, routing-relevant properties rather than superficial domain labels, ensuring generalized auto-delegation. Third, a curated analysis of twelve critical adjacent systems is presented, isolating evidence-backed mechanisms OCC can adopt and structural limitations OCC must improve upon. Fourth, the architecture of the proposed task-fact contract is outlined, demonstrating how uncertainty can be natively represented in the system's internal currency. Finally, the analysis delivers decision-ready conclusions, separating the absolute minimum essential routing facts from deferrable optimizations, and establishing the foundation for the subsequent integration of the OCC Capability Registry and deterministic routing policy frameworks.1. What OCC Needs to Know About a TaskTo function as a deterministic control layer, OCC must extract a highly structured profile of every inbound request prior to execution. The following elements represent the critical task facts the router must isolate to construct a dependable operational strategy.Primary GoalPlain-English Name: Primary Goal
What it means: The fundamental objective of the user's request. It captures the semantic intent and the overarching purpose of the computational workload, essentially answering "what" the system is attempting to achieve.
Why it changes a routing decision: The primary goal dictates the broad classification of the task, influencing which capabilities must be queried from the registry. A goal centered on data extraction will prioritize models with high context-window retention, whereas a goal centered on logic synthesis will prioritize frontier tier architectures. Furthermore, it serves as the foundational payload for delegation, acting as the "Commander's Intent" passed to sub-agents.
Requirement Level: Required.
Handling when unknown/unclear: If the primary goal cannot be mathematically mapped or semantically understood, the OCC router must halt the execution pathway. It must generate an ambiguity_flag and initiate a conversational elicitation loop, asking the user to clarify their objective before consuming any execution tokens.Expected Result StructurePlain-English Name: Expected Result Structure
What it means: The exact termination condition, output format, or side effect expected at the conclusion of the task. This defines the physical or digital manifestation of the goal, such as a compiled binary, a summarized text file, a committed repository, or a simple conversational reply.
Why it changes a routing decision: Understanding the expected result alters routing by establishing the criteria for task completion and evaluation. If the expected result is highly structured (e.g., a strictly formatted JSON payload), the router must provision a secondary evaluator model capable of verifying that format. It also dictates tool binding; an expected result of a "modified file" mathematically guarantees the need for file-write permissions.
Requirement Level: Required.
Handling when unknown/unclear: When the expected result is unclear, OCC must not hallucinate an output format. Instead, it must default the task to an unconstrained_chat mode, routing the response directly back to the user interface as plain text, thereby shifting the burden of structuring the final output back to the user in subsequent prompts.Execution UrgencyPlain-English Name: Execution Urgency
What it means: The strictness of the timeline for task completion. It measures the time-decay of the value of the output, distinguishing between tasks that require immediate milliseconds-level latency and tasks that can be fulfilled hours later.
Why it changes a routing decision: Urgency directly alters the weighting between latency, quality, and cost. High-urgency tasks must bypass complex planning stages and route directly to fast, high-availability endpoints. Low-urgency tasks unlock massive cost optimizations; they can be delayed until off-peak API hours, routed to slower local-compute clusters, or scheduled via Slurm-style backfill algorithms to utilize idle system resources.
Requirement Level: Required.
Handling when unknown/unclear: If urgency is explicitly unknown, the system must inherit the status of the user. If the user presence is synchronous, urgency defaults to "moderate." If the user is offline, urgency defaults to "low," enabling deferred execution processing.User Presence and AvailabilityPlain-English Name: User Presence and Availability
What it means: A boolean or state-based representation indicating whether the user is actively waiting at the interface (synchronous) or has stepped away/gone to sleep (asynchronous).
Why it changes a routing decision: User presence acts as a massive constraint on blocking operations. If a user is present, the router must prioritize rapid, streaming feedback to maintain a perception of system responsiveness. If the user is absent, the router is freed from latency constraints, allowing it to execute deep, multi-round iterative loops, heavy research tasks, or parallel agentic swarms without freezing a user-facing interface.
Requirement Level: Required.
Handling when unknown/unclear: When user availability cannot be confirmed via system heuristic signals (e.g., IDE activity, mouse movement), OCC must adopt a conservative posture and assume the user is present (true). This prevents the system from accidentally taking hours to complete a simple task simply because the presence detector failed.Output Quality ThresholdPlain-English Name: Output Quality Threshold
What it means: The acceptable boundary for logic errors, hallucinations, or lack of depth. It defines whether the user requires absolute precision (e.g., critical production code) or is willing to accept approximate heuristics (e.g., brainstorming ideas).
Why it changes a routing decision: Quality needs trigger a direct tradeoff calculation against cost and speed. A task requiring maximum output quality will bypass smaller local models entirely, routing directly to the highest-tier frontier capabilities. Conversely, a low-quality-threshold task acts as an optimization vector, allowing OCC to route to extremely cheap, open-weight models, preserving the budget for more complex workloads.
Requirement Level: Required.
Handling when unknown/unclear: When quality requirements are unstated, OCC must default to a mid-tier capability routing. This prevents wasting premium tokens on mundane tasks while ensuring the output does not suffer from the severe degradation associated with the lowest-tier models.Budget SensitivityPlain-English Name: Budget Sensitivity
What it means: The financial constraints applied to the compute cycle for a specific request. It dictates the maximum acceptable cost threshold for task completion.
Why it changes a routing decision: Budget sensitivity operates as a gating mechanism during the provider selection phase. A high budget sensitivity enforces a strict preference for local-first compute, owned hardware, or lower-tier API providers. It fundamentally overrides quality parameters; if a task requires high quality but is highly budget-sensitive, the router must optimize for the best available open-weight model rather than an expensive proprietary endpoint.
Requirement Level: Optional.
Handling when unknown/unclear: If budget sensitivity is not explicitly extracted from the prompt, the router must fall back to the user's global configuration settings stored within OCC. If no global setting exists, it defaults to a balanced cost-to-performance ratio.Environmental PermissionsPlain-English Name: Environmental Permissions
What it means: The authorization and access level required to execute the task on the host machine or connected networks (e.g., read-only sandbox, write-access to a specific repository, root terminal access).
Why it changes a routing decision: Permissions act as hard constraints, akin to Kubernetes taints and tolerations. A task cannot be routed to an agent or model that lacks the requisite local environment permissions. Identifying permission requirements early ensures that OCC does not delegate a file-writing task to an isolated, read-only container, thereby preventing mid-execution state failures.
Requirement Level: Conditional (Only required if environmental interaction is detected).
Handling when unknown/unclear: When permission needs are ambiguous, the router must isolate the task in the most restrictive environment available (e.g., an ephemeral, read-only sandbox). Permission escalation must be requested from the user dynamically only when the executing model encounters a hard barrier.Required Tools and File DependenciesPlain-English Name: Required Tools and File Dependencies
What it means: The specific external capabilities (e.g., web search, Python interpreter, SQL connector) and the exact context documents or directories required to process the request.
Why it changes a routing decision: This fact maps directly to the Capability Registry. If a task requires web search, models without internet access are mathematically excluded from the routing pool. Furthermore, identifying file dependencies allows the router to pre-fetch context, loading the necessary documents into memory before initiating the model inference cycle, drastically reducing latency.
Requirement Level: Conditional (Only required if the prompt implies external data needs).
Handling when unknown/unclear: If the system cannot determine the required tools, it must route the task to a generalized model with access to a basic suite of discovery tools (e.g., ls, grep, basic web search) so the model can self-discover the necessary context during execution.Task SplittabilityPlain-English Name: Task Splittability
What it means: An architectural evaluation of whether the task can be parallelized into independent sub-tasks, or if it must be executed strictly sequentially within a single context window.
Why it changes a routing decision: Splittability dictates the entire delegation topology. A highly splittable task (e.g., "Summarize these 10 distinct PDFs") can be divided among a swarm of specialized, lower-cost models operating in parallel, significantly reducing total latency. Tightly coupled tasks (e.g., "Refactor this recursive algorithm") must remain monolithic to prevent fatal state desynchronization and context fragmentation.
Requirement Level: Optional.
Handling when unknown/unclear: When unknown, OCC must conservatively treat the task as a monolithic, sequential operation. Splitting an inherently coupled task results in catastrophic hallucinations, as the parallel nodes will lack the necessary shared latent context to complete their specific branches.Risk and ReversibilityPlain-English Name: Risk and Reversibility
What it means: Risk quantifies the potential damage of a system error or hallucination. Reversibility determines how easily an action can be undone (e.g., drafting a local markdown file is highly reversible; executing a DROP TABLE command on a remote database is irreversible).
Why it changes a routing decision: High-risk, irreversible tasks drastically alter routing by overriding all automation protocols. They enforce mandatory pre-execution checks, dry runs, and the assignment of the highest-tier reasoning models for secondary validation. The risk level also determines the priority queue positioning; critical risk interventions jump ahead of standard background tasks.
Requirement Level: Required.
Handling when unknown/unclear: If risk or reversibility is unclear, OCC must natively default to a critical risk classification. The architecture must operate under the strict assumption that any unverified or ambiguous action involving external systems is a destructive action.Human Approval NeedsPlain-English Name: Human Approval Needs
What it means: Defines whether the task requires explicit human authorization before execution begins, at critical junctures during a multi-step process, or before final deployment.
Why it changes a routing decision: Approval needs convert a standard sequential task into a durable, paused state machine. If approval is needed, the router must schedule the task to run up to the boundary condition, halt execution, preserve the state in memory, and generate a notification payload for the user interface, resuming only upon cryptographic confirmation.
Requirement Level: Required.
Handling when unknown/unclear: When the requirement for human approval is unstated, it must inherit the state of the Risk and Reversibility facts. If the task is low risk, approval defaults to false. If the risk is moderate or above, approval defaults to true.Semantic AmbiguityPlain-English Name: Semantic Ambiguity
What it means: A measurement of the clarity of the instructions and the presence of all necessary variables. It tracks the entropy of the user's prompt.
Why it changes a routing decision: If ambiguity is high, routing the task to a reasoning model wastes API tokens and invites unpredictable behavior. High ambiguity forces the router to shift from an execution pathway to an elicitation pathway, invoking a cheap, conversational model explicitly prompted to ask clarifying questions until the ambiguity index drops below a designated threshold.
Requirement Level: Required.
Handling when unknown/unclear: If the ambiguity of the prompt itself cannot be measured, the router must pass the prompt to a fast, low-cost classifier specifically trained to score information completeness before allowing it to reach the main routing engine.Privacy and Data SensitivityPlain-English Name: Privacy and Data Sensitivity
What it means: The confidentiality level of the data involved in the task, identifying the presence of personally identifiable information (PII), proprietary source code, or confidential financial metrics.
Why it changes a routing decision: A high privacy requirement changes the routing decision by strictly limiting execution to local-first compute environments or enterprise-contracted APIs with guaranteed, legally binding zero-retention policies. It acts as an immutable firewall, preventing sensitive data from leaking into public model training pipelines or unauthorized geographic regions.
Requirement Level: Conditional (Required when heuristics detect sensitive data patterns).
Handling when unknown/unclear: If privacy sensitivity is unknown, OCC should evaluate the global system defaults. If deployed in an enterprise or secure local context, all data must be treated as highly sensitive by default, restricting all routing to local models until explicitly overridden by the user.Likely Work SizePlain-English Name: Likely Work Size
What it means: A computational prediction of the token volume, temporal duration, and memory intensity required to complete the task from start to finish.
Why it changes a routing decision: Work size prediction alters temporal scheduling. Massive, high-duration tasks can be sent to backfill schedulers or delayed until optimal compute availability is reached, preventing short, urgent tasks from being blocked behind massive processing queues. It also dictates memory allocation for local compute clusters.
Requirement Level: Optional.
Handling when unknown/unclear: When the work size cannot be confidently estimated, the router must assign a dynamic, highly elastic timeout window and monitor the task actively, ready to preempt or pause the process if it begins consuming disproportionate system resources.Inter-Task DependenciesPlain-English Name: Inter-Task Dependencies
What it means: A mapping of prerequisite conditions. It defines whether the current task relies on the output of a currently running process, a pending file download, or a prior state transition.
Why it changes a routing decision: Dependencies govern workflow orchestration. A task with unresolved dependencies cannot be routed to an execution model; it must be routed to a pending queue within a durable state machine (like Temporal) until the upstream tasks report successful completion.
Requirement Level: Conditional (Only for multi-part auto-delegated sub-tasks).
Handling when unknown/unclear: If dependencies are unclear, the task must be treated as dependent on the immediately preceding user action, enforcing a strict sequential execution order to preserve causal integrity.2. Decision-Ready ConclusionsThis research provides the foundational logic and structural blueprints for the Overseer Command Center's routing architecture. By studying advanced systems in medical triage, high-performance distributed computing, and military command, the specific mechanisms required for robust, safe, and efficient AI orchestration have been isolated.The Smallest Set of Essential Task FactsOCC cannot operate reliably or safely without extracting a core, non-negotiable subset of task facts. The smallest set of facts required to prevent catastrophic routing failure includes:Goal and Expected Result: Without these, the system cannot know when to stop generating or how to evaluate success.Urgency and User Presence: Without these, the system cannot manage time effectively, risking massive API costs on deferred tasks or blocking the user interface during synchronous work.Risk Level and Privacy Constraints: Without these, the system represents an active danger, risking destructive file system modifications or the leaking of proprietary local code to public LLM training pipelines.Required Tools: Without this, tasks will be routed to sandboxed models incapable of fulfilling the request, resulting in guaranteed execution failure and endless retry loops.Task Facts That Are Useful But Can WaitConversely, several facts are highly useful for long-term optimization but can be deferred for later iterations of the OCC architecture without compromising system safety:Work Size Estimation: While highly valuable for maximizing cluster throughput via Slurm-style backfill algorithms, early versions of OCC can function securely by treating all tasks dynamically and utilizing simple timeout thresholds.Splittability: Advanced parallelization and swarm topology generation are complex. Initially, OCC can enforce sequential, monolithic execution for all tasks to guarantee state stability before introducing multi-agent branching.Budget Sensitivity: Dynamic, per-task budget extraction can be delayed by relying entirely on static, global user preferences to govern cost-based routing decisions.Unresolved Questions Requiring Later ResearchBefore final architectural decisions regarding the router implementation are locked, several critical technical questions remain unresolved:Latency Overhead: What is the exact latency cost of the classification step required to generate the Task Fact Contract? If extracting these facts via a semantic router or a lightweight classifier model introduces multiple seconds of delay, it may negatively impact the user experience for simple, highly interactive tasks.Zero-Shot Accuracy: What is the empirical accuracy of small, local LLM-based classifiers in correctly predicting tool dependencies and risk levels from highly complex, multi-layered prompts? Can a quantized 8B local model reliably generate the Task Fact Contract without hallucinating constraints?Specific Findings Feeding the Next Two Research RunsThese architectural findings dictate the immediate structural requirements for the next two phases of OCC research :Capability Registry Evidence: The registry must be designed specifically to mathematically intersect with the proposed Task Fact Contract. It cannot operate as a simple list of API keys. The registry must track model capabilities as Kubernetes-style "tolerations" (e.g., has_web_access: true, is_local_compute: true, max_context_window: 128000) and strictly define cost-per-token metrics. This allows the router to instantly discard models that lack the required tools or privacy compliance demanded by the contract.Deterministic Routing Policy: The routing policy itself must be structured as a multi-stage funnel rather than a single black-box LLM prompt. It must first apply hard boolean constraints (eliminating models that fail tool, privacy, or risk requirements). Second, it must calculate queue priority using an ITIL-style algorithm ($Priority = Urgency \times Risk$). Finally, it must apply dynamic weighted scoring based on budget, model fit, and output quality to select the optimal surviving capability for execution.Evidence Needed Before Architecture DecisionsBefore OCC commits to a specific classifier framework or storage engine to handle the Task Fact Contract, empirical benchmarking is required. OCC must benchmark small, local models (such as Llama-3-8B or equivalent open-weight architectures) against the Task Fact Contract schema to determine their speed and JSON-structuring accuracy compared to API-based models. Evidence must definitively prove that the computational overhead of the facts-extraction phase is mathematically offset by the cost and time savings generated by correct, deterministic routing.By implementing this structured, fact-driven approach, OCC will evolve beyond a simple pass-through API orchestrator.
