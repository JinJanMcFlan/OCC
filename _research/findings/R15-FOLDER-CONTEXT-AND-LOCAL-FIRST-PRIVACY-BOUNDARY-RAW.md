A6 — Folder Context and Local-First Privacy Audit [OCC]The integration of local filesystem context into AI orchestration platforms presents a critical security and privacy paradigm. As the landscape of multi-model, agentic workflows expands, the intersection between local project files and external large language models (LLMs) has exacerbated the risks of unintended data exfiltration, prompt-injected secret harvesting, and privilege escalation. For the Overseer Command Center (OCC) to maintain its deterministic, provider-neutral, and local-first architecture, the application must establish a rigorous perimeter around project-folder binding. This build-accelerating research audit investigates the minimum project-folder context that a local-first desktop OCC may safely gather, display, and use for task routing and packaging. It establishes the boundary conditions required to prevent indiscriminate filesystem scanning, unnecessary exposure of sensitive configurations, unsafe external sharing, and deceptive user consent models. All factual claims regarding operating-system permissions, Electron framework security, and desktop threat models reflect the current technical state verified as of June 21, 2026. This analysis directly feeds OCC's safe folder-binding implementation and Base UX context planning, without prematurely locking a final storage architecture or filesystem scanner implementation, ensuring that the D1 Router Build Package remains unblocked.Existing Folder-Binding, Permission, and Local-Context PatternsThe current technological landscape of desktop applications and AI-assisted integrated development environments (IDEs) relies on a complex amalgamation of operating-system-level sandboxing, application-framework restrictions, and user-facing trust models. Understanding these mechanisms is paramount for OCC to construct a resilient folder-context envelope. The dominant patterns operate across the operating system layer, the application framework layer, and the editor-specific security layer.The foundation of modern desktop security rests upon strict isolation protocols mandated by operating systems, particularly for applications distributed via official storefronts or those handling sensitive user data. On macOS, the operating system isolates applications using the App Sandbox and the Transparency, Consent, and Control (TCC) framework. Sandboxed applications are categorically denied arbitrary read or write capabilities to user directories such as the Desktop, Documents, or Downloads folders. Access within this ecosystem is inherently intent-based. When a user interacts with a native file picker, such as the NSOpenPanel, the system’s un-sandboxed Powerbox API generates a temporary sandbox extension token. This kernel-issued token grants the requesting application read or write access to the explicitly selected file or folder, effectively poking a controlled hole in the sandbox barrier.However, IDEs and orchestration platforms like OCC require persistent access across application restarts to function effectively as project workspaces. To facilitate this, macOS provides security-scoped bookmarks. When an application requests persistent access, a system daemon known as the ScopedBookmarkAgent generates a cryptographically signed HMAC-SHA256 blob. This blob binds the targeted file path to the application's unique code signature and a cryptographic key. To utilize this bookmark in subsequent sessions, the application must explicitly invoke the startAccessingSecurityScopedResource() method before accessing the file, and crucially, it must call stopAccessingSecurityScopedResource() immediately upon completion. Failure to balance these calls results in severe kernel resource leaks, silently stripping the application of its ability to reach outside the sandbox until a full restart occurs.In contrast, the Windows operating system employs the AppContainer isolation environment, primarily utilized for Universal Windows Platform (UWP) and MSIX-packaged applications. The AppContainer architecture enforces comprehensive credential, device, file, network, process, and window isolation based on the principle of least privilege. By default, sandboxed Windows applications cannot access the broader filesystem beyond their designated application data directories. Broader access requires developers to declare explicit capabilities within the application manifest. The broadFileSystemAccess capability, for instance, triggers heightened scrutiny during Microsoft Store onboarding and forces a user-facing consent prompt. Alternatively, applications can rely on the Windows File Picker to capture user intent and temporarily broker access to selected directories, mirroring the macOS intent-based approach. Furthermore, Windows Sandbox environments can be explicitly configured via .wsb files to map specific host folders into the isolated container, providing granular control over read and write permissions at the virtualization boundary.Because OCC is constructed upon an Electron Forge, Vite, React, and TypeScript stack, the application inherently inherits Electron’s specific architectural boundaries and communication paradigms. Between the years 2024 and 2026, Electron’s security defaults matured significantly to mitigate remote code execution (RCE) and cross-site scripting (XSS) vulnerabilities. Modern Electron applications enforce strict process separation. By default, the webPreferences configuration mandates contextIsolation: true, nodeIntegration: false, and sandbox: true for all renderer processes. This architecture ensures that web-based JavaScript running in the UI cannot directly access underlying Node.js APIs, such as the filesystem module (fs) or the child process execution module (child_process).Inter-Process Communication (IPC) serves as the sole conduit between the untrusted renderer process and the privileged main process. This communication is brokered exclusively via a preload script utilizing contextBridge.exposeInMainWorld(). Secure applications treat all incoming IPC messages as untrusted external input, deploying strict schema validation to prevent path traversal attacks and arbitrary command execution. Furthermore, to serve local files to the renderer securely, modern Electron applications have abandoned the insecure file:// protocol. Instead, they register privileged custom protocols, such as app://, using the protocol.registerSchemesAsPrivileged() API. This architectural shift allows the main process to enforce strict path validation, preventing directory traversal exploits while successfully bypassing Content Security Policy (CSP) limitations that would otherwise block local media rendering.Beyond the operating system and framework levels, applications implement domain-specific trust models to manage local context. Visual Studio Code (VS Code) implements a prominent "Workspace Trust" boundary. When a user opens an unfamiliar folder, VS Code displays a blocking modal asking if the user trusts the authors of the files. If deemed untrusted, the workspace enters "Restricted Mode." This mode aggressively curtails automatic code execution by disabling tasks, debugging capabilities, workspace-specific settings defined in .vscode/settings.json, integrated terminals, and AI agent functionalities. The rationale is that simply opening a folder should not trigger malicious build scripts or allow an extension to execute arbitrary binaries.With the proliferation of AI coding assistants, such as Cursor, Windsurf, and GitHub Copilot, traditional version control exclusion mechanisms like .gitignore proved fundamentally insufficient for privacy management. AI tools frequently require access to git-ignored files, such as build logs or local configuration overrides, to provide accurate context, but they must simultaneously be blocked from reading sensitive credentials. Consequently, a parallel ecosystem of AI-specific ignore files emerged, including .cursorignore, .copilotignore, and .aiexclude. OCC must actively mitigate these specific vulnerabilities, which span overcollection, psychological warning fatigue, and deep systemic sandbox escapes.The most critical failure mode in current AI orchestration is the indiscriminate reading of environment variables and credential files, leading to catastrophic secret exposure. AI agents and orchestrators natively seek maximal context to fulfill complex user requests accurately. If an OS-level permission grants an application broad access to a folder, the application's internal parsers may automatically scan sensitive configuration files, such as .env, .pem, credentials.json, or SSH keys. Standard .gitignore files, designed strictly to prevent version control commits, offer zero protection against local AI reads. Consequently, if an AI router sweeps a directory to build a contextual understanding of a project, raw API keys, database passwords, and production secrets are ingested into the LLM context window. Depending on the specific provider and the user's data retention agreements, these secrets may be logged, utilized for future model training, or stored indefinitely in accessible chat histories.This overcollection vulnerability is exponentially compounded by the threat of prompt injection exfiltration. If a malicious actor places a crafted instruction inside a seemingly benign file—such as a cloned repository's README.md, a downloaded open-source package, or even a deep code comment—the local AI parsing the file to gather context may inadvertently execute the hidden prompt. If the AI tool possesses both filesystem access and network capabilities, the injected prompt can instruct the AI to read the local .env file and silently exfiltrate the contents via an HTTP POST request to an external server controlled by the attacker. This attack vector requires no explicit user interaction beyond opening the poisoned repository with an AI-enabled IDE.Binary trust models, such as VS Code's Workspace Trust, suffer from severe human-factors degradation, leading to unclear consent and warning fatigue. The prompt asking if the user trusts the authors of the files acts as a monolithic, all-or-nothing gate. Because the "Restricted Mode" aggressively breaks core IDE functionality—including integrated terminals, formatting tools, language servers, and task runners—users are highly incentivized to reflexively click "Trust" simply to make the application functional, effectively bypassing the security control without conducting any meaningful inspection of the underlying code. Furthermore, this binary model lacks necessary granularity. Trusting a workspace grants the application the right to execute tasks, but it fails to differentiate between local, deterministic execution (such as running a local Python linter) and remote, non-deterministic transmission (such as sending proprietary source code to a cloud-based LLM provider for refactoring).Operating system-level persistent permissions are also fragile and subject to logical corruption, resulting in stale context and potential privilege escalation. On macOS, security-scoped bookmarks can become corrupted or resolve unpredictably. Complex interactions with external volumes, particularly those utilizing third-party drivers like NTFS for Mac, have been documented causing security-scoped bookmarks to resolve incorrectly to the root directory (/) instead of the intended external path, creating massive security and usability flaws. Furthermore, the intricate nature of the startAccessingSecurityScopedResource and stopAccessingSecurityScopedResource lifecycle leads to silent access revocations if an application crashes or fails to execute the stop command, requiring the user to navigate the TCC prompt matrix repeatedly.More alarmingly, sophisticated sandbox escapes continue to exploit vulnerabilities in how operating systems validate these persistence tokens. The CVE-2025-31191 vulnerability, colloquially known as "Sploitlight," demonstrated that malicious sandboxed applications could exploit improper state management in the handling of security-scoped bookmarks. By leveraging vulnerabilities in Spotlight importer plugins and manipulating the HMAC-SHA256 authentication blobs generated by the ScopedBookmarkAgent, attackers could forge or replay bookmarks. This allowed them to escape the App Sandbox boundary entirely and access files protected by TCC, such as the Downloads folder or Apple Intelligence caches, without any user interaction or consent beyond the initial application launch.Finally, the phenomenon of unsafe external sharing is exacerbated by deceptive user interfaces. Applications frequently market themselves as "local-first" or privacy-centric while failing to clearly define their telemetry and routing boundaries. If a user binds a highly sensitive local folder to an orchestrator, they may operate under the assumption that all subsequent processing utilizes local hardware or local models. If the orchestrator silently routes a complex task to a remote, closed-cloud provider—perhaps due to a local model's context window limit being exceeded or an automatic fallback mechanism triggering—the local proprietary code is unsafely shared externally without a secondary, explicit consent step, violating the foundational premise of local-first privacy. Relying on users to perfectly configure .cursorignore files or expecting LLMs to possess the "common sense" to ignore API keys is a failing strategy.The candidate OCC improvement hypothesis posits that superior local-first security can be achieved by abandoning monolithic workspace trust in favor of an Intent-Driven, Layered Context Envelope that strictly segregates topological metadata from semantic file content. By enforcing hardcoded, operating-system-level IPC hooks that globally and immutably block secret-bearing files before transmission, and by integrating this layered envelope directly into the OCC R3 Split-Gate deterministic router architecture, OCC can safely route bounded tasks. This approach will prevent the exposure of local secrets to prompt injection attacks, eliminate the psychological warning fatigue associated with binary trust models, and remove the burden of manual exclusion configuration from the user.This hypothesis asserts that security must be enforced deterministically at the orchestration communication layer. The router must be able to understand the structure of a project to make intelligent delegation decisions without prematurely reading the vulnerable contents of the files themselves.Proposed Minimum Folder-Context EnvelopeTo operationalize the candidate hypothesis and secure the application boundary, OCC must implement a highly specific minimum folder-context envelope. This envelope defines the precise parameters governing what data the OCC Router and the Research Engine may extract, hold in memory, and utilize for decision-making. The envelope is divided into strict layers: Default Collection, Explicit Approval, the External Network Boundary, and Revocation Handling.Default Collection: The Topology LayerWhen a user explicitly binds a project folder to OCC via a native operating system file dialog, the application enters the Topology Layer. This is the default state of collection and requires no further user interaction beyond the initial folder selection.In this state, OCC is explicitly authorized to collect structural metadata. This includes file paths, file names, file extensions, file sizes, and standard filesystem timestamps indicating creation and modification dates. Crucially, OCC must automatically ignore large dependency directories such as node_modules, standard build artifact directories like /dist or /build, version control directories like .git, and binary media files, unless the R1 General Router Task Facts explicitly demand awareness of their structure.The primary purpose of the Topology Layer is to feed the R1 task extraction process. By mapping the structure of the bound folder, the OCC router can answer vital logistical questions necessary for deterministic routing. It allows the system to determine if the task involves a massive multi-package monorepo, if critical architectural dependencies are present, or if the estimated aggregate file size will exceed the context window constraints of a locally hosted execution model.The ironclad constraint of the Topology Layer is that the semantic contents of the files are never read into memory. The topological data may be freely utilized to calculate routing schedules, estimate task complexity, and determine model fit against the R2 Capability Registry, but no semantic source code is processed, entirely mitigating the risk of passive secret exposure or prompt injection during the initial assessment phase.Explicit Approval: The Semantic LayerWhen an active, bounded task explicitly requires reading or modifying specific source code files to fulfill the user's objective, OCC transitions into the Semantic Layer.This transition is triggered exclusively when the R3 Deterministic Router determines that semantic context is an absolute prerequisite for task execution. At this juncture, OCC is permitted to read the specific file contents into memory. However, to combat the .env vacuum effect, OCC's IPC layer must implement a hardcoded, un-bypassable blocklist for secret-bearing files.This Ironclad Blocklist dictates that files matching patterns such as .env*, *.pem, *.key, credentials.json, id_rsa, or any file residing within a .ssh directory must never be read into the orchestrator's semantic memory under any circumstances. This system-level blocklist takes absolute precedence over any user-defined configuration or AI request. If an executing agent or a specific bounded task absolutely requires access to credentials to perform an operation (e.g., deploying to a cloud environment), those secrets must be injected at runtime via an isolated memory mechanism—such as environment variables passed strictly and exclusively to a child execution process—and never passed through the OCC orchestrator's primary semantic memory or logging mechanisms.The External Network BoundaryA critical component of the OCC context envelope is the strict architectural distinction between local task execution and remote capability assignment. This boundary leverages the capabilities defined in the R2 Capability Registry to enforce data sovereignty.The inviolable rule of the External Network Boundary is that semantic context—specifically, strings representing source code or file contents extracted from a bound local folder—must never leave the physical device unless two conditions are met: the R3 Deterministic Router has algorithmically selected a remote capability provider as the optimal route, and the user has explicitly authorized external transmission for that specific capability class.This rule directly interacts with the proposed R3 Split-Gate architecture. If a user requests a complex refactoring task on a bound local folder, and the highest-scoring model capable of handling the request is a closed-cloud API, the R3 post-selection commitment check must intervene before execution begins. The OCC user interface must explicitly halt the queue and state the routing rationale: "This route requires sending 4 local files to [Remote Provider] to fulfill the output quality requirements. Approve?". If the user declines this transmission, the router must deterministically fall back to local capabilities, even if it results in lower output quality, or mark the task as blocked due to safety constraints. This eliminates the deceptive UI failure mode and ensures continuous user consent for data exfiltration.Handling Stale or Revoked ContextFolder bindings within OCC are temporal entities. A bound folder may be deleted by the user, an external APFS or NTFS drive may be abruptly disconnected, or macOS TCC permissions may be reset during a system update.OCC must intercept these filesystem exceptions gracefully and securely. They are documented here to prevent immediate scope creep and preserve the D1 build timeline, while capturing high-value operational enhancements for subsequent Research Engine cycles.Semantic Circuit Breakers (Dynamic Payload Analysis)
While hardcoded blocklists protect known secret file structures (like .env), sophisticated prompt injection attacks could instruct an AI to print runtime secrets into a benign-looking .txt file or interleave them within standard JSON output, effectively bypassing extension-based checks. OCC could implement a local, low-latency regex or entropy-based scanner acting as a "Semantic Circuit Breaker" on the output stream. If the orchestrator detects high-entropy strings resembling API keys or private keys traversing from the executing model toward the external network interface, the connection is instantly severed, and a security exception is raised. This remains parked, requiring significant validation regarding false-positive rates and processing overhead.Content-Addressed Zero-Trust Hashes
Relying solely on file paths for capability assignment and context gathering is fragile. If a file is modified externally by the user or another process during a routing queue wait state, the router may execute a task based on stale assumptions. The OCC topology layer could calculate lightweight cryptographic hashes (such as BLAKE3) of relevant files during the R1 task ingestion phase. If the file's hash changes before the R3 execution gate opens, the router flags the context as compromised and forces a re-evaluation of the routing factors. This concept currently falls under the deferred architecture boundaries of the R5 Shared Records audit but merits ongoing monitoring.Native Zero-Disk Secret Management
The root vulnerability of AI context reading is the physical presence of secrets on the disk within the project directory. OCC could integrate a native secure enclave or interface with dedicated secret management solutions (e.g., Bitwarden Secrets Manager, Infisical) to broker credentials directly into memory exclusively for authorized child execution processes. This completely removes the need for .env files to exist in the filesystem, fundamentally nullifying the risk of AI exfiltration via local file scanning. This concept aligns perfectly with OCC’s cognitive consolidation hypothesis and should be escalated for evaluation post-D1 as a core feature of the OCC execution sandbox environment.
