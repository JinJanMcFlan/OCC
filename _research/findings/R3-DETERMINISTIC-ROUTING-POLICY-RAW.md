Research Report: Deterministic Routing Policy and Scheduling for the Overseer Command Center (OCC)Founder BriefThe Overseer Command Center (OCC) requires a deterministic, provider-neutral routing architecture capable of translating structured task requirements and capability evidence into repeatable, explainable execution decisions. The primary objective of this research is to define the exact mechanisms by which the OCC router will evaluate non-commensurate variables—such as budget constraints, urgency, user presence, and model fit—without relying on non-deterministic large language model inferences. The resulting system must operate as a stable, predictable control layer across a shifting landscape of underlying tools, external APIs, and local compute resources.A central architectural conflict identified in earlier research phases concerns the precise ordering of the routing pipeline, specifically the placement of weighted scoring versus hard-rule gates. The analysis establishes that this is not a true architectural conflict requiring a redesign, but rather a misclassification of the gating mechanism into a single monolithic layer. Borrowing from highly scalable distributed resource schedulers such as Kubernetes and Google Borg, the solution mandates a split-gate approach. Hard restrictions must operate both as a pre-scoring eligibility filter—to eliminate capabilities that fundamentally lack the authorizations or technical capacity to perform the work, thereby saving computational overhead—and as a post-scoring operational gate to enforce dynamic constraints, such as real-time budget caps or human approval holds, on the winning route prior to final execution.To make repeatable decisions among the remaining eligible candidates, the research indicates that OCC must utilize a structured Multi-Criteria Decision Analysis (MCDA) framework. Specifically, the implementation of Simple Additive Weighting (SAW) will resolve the complex interplay between OCC’s five locked routing factors: Cost, Urgency, User Presence, Model Fit, and Output Quality Tier. By normalizing these disparate factors into a standardized dimensional scale and applying user-defined prioritization weights, the router mathematically determines the most appropriate capability. Furthermore, Urgency (the time sensitivity of the task's value) and User Presence (the availability of the human to review or guide the work) are preserved as strictly independent variables. This separation ensures that urgent, autonomous background tasks are routed fundamentally differently than low-urgency tasks requiring active human oversight.Scheduling and deferral decisions within OCC will draw from Airport Collaborative Decision Making (A-CDM) frameworks and the Saga distributed architectural pattern. A-CDM principles dictate that system delays should be absorbed in low-cost states—such as remaining queued or deferred to off-peak batch windows—rather than during active execution. When capabilities fail mid-execution or unresolvable uncertainties arise, OCC will utilize Saga principles to execute compensating transactions, safely reverting side-effects rather than blindly retrying failed operations. When facts are uncertain, OCC will adopt Risk-Based Authentication (RBA) concepts to dynamically request step-up clarification from the user rather than halting entirely or guessing.The following report provides the comprehensive, provider-neutral routing decision process, a detailed matrix for scheduling and deferral policies, robust handling of uncertainty and fallback behaviors, a deep analysis of fourteen adjacent operational systems, and the decision-ready conclusions required to implement the OCC deterministic router. The design preserves OCC's provider neutrality, ensures all decisions are highly auditable for graph-based visualization, and rejects the notion that complex orchestration requires unpredictable routing logic.A Practical Provider-Neutral Routing Decision ProcessThe routing decision process transitions a task from a set of structured facts to a finalized execution route. To ensure predictability, auditability, and efficiency, the process is structured as a sequential pipeline that strictly separates hard restrictions, normative tradeoffs, scheduling logic, and fallback behaviors. The pipeline does not rely on a large language model to guess the appropriate route; instead, it relies on structured data evaluation.The practical execution flow consists of the following distinct phases.The ingestion and fact extraction phase initiates the process. The router receives the structured task facts from the Classifier (established in prior research) and queries the current capability evidence from the Capability Registry. This combination forms the foundational dataset for the routing decision. If missing critical information is detected immediately—such as an ambiguous goal or undefined privacy requirements—the routing process suspends. The system acts on the principle of Risk-Based Authentication, treating missing context as a high-risk scenario, and triggers a clarification request to the user before proceeding.The pre-scoring eligibility filtering phase represents the first application of hard restrictions. Before any comparative scoring occurs, the router acts as a strict boolean filter. Using mechanisms similar to the Kubernetes scheduler's preFilter and filter plugins, OCC evaluates the task facts against the Capability Registry. Any capability that lacks the required tool access, file permissions, privacy compliance, or input context capacity is immediately eliminated from the candidate pool. This ensures OCC never wastes compute cycles scoring impossible routes and prevents a highly capable but unauthorized capability from skewing the normalization algorithms used in later scoring.The capability cost estimation phase processes the surviving eligible candidates. The Cost Layer calculates the estimated execution cost based on the task's expected size, complexity, and the registry's current pricing evidence. If a candidate's estimated cost exceeds the user's hard budget cap for the task, it is eliminated from the pool.The multi-criteria tradeoff scoring phase evaluates the remaining valid candidates. OCC passes the candidates to the Weighted Scoring layer, utilizing a deterministic Simple Additive Weighting (SAW) algorithm. The algorithm evaluates how well each candidate aligns with the user's preferences across the five routing factors. Candidates receive a normalized score based on these factors and the user's configured weights. The candidate with the highest aggregate score is selected as the primary route, representing the optimal mathematical intersection of the user's priorities and the available capabilities.The timing and scheduling evaluation phase determines when the primary route should be executed. Once a route is selected, the Scheduler evaluates the task's urgency factor, the selected capability's current availability, and potential peak-hour pricing windows. Drawing on concepts from workload managers like Slurm and HTCondor, the Scheduler outputs a precise timing decision. This decision dictates whether the task will run immediately, enter a queue for capacity, defer to a low-cost batch window, or hold for an upstream dependency.The human approval gating phase acts as the final hard restriction before commitment. If the task facts dictate high risk, strict reversibility limits, or if the selected capability has an unverified privacy status, the routing decision halts at an Approval Gate. The system logs the intended route and blocks execution until explicit human authorization is recorded, ensuring the user retains ultimate authority over sensitive operations.The post-scoring verification and commitment phase finalizes the process. In the final millisecond before execution, the router performs a state check to verify that the selected local machine or remote API has not gone offline since the routing decision began. The complete decision—including rejected alternatives, applied weights, and the rationale—is committed to the Decision Log as structured JSON to enable future auditing and graph visualization.Interplay of the Five Routing FactorsOCC evaluates routing tradeoffs using five independent factors. To calculate a route deterministically, OCC normalizes the raw data of each capability into a dimensionless scale from zero to one for each factor, then multiplies these by the user's preferred weightings. This methodology is derived from Multi-Criteria Decision Analysis (MCDA), specifically Simple Additive Weighting (SAW).The definitions and mechanical interactions of these factors are as follows.Cost represents the estimated financial expenditure or local compute resource utilization required to complete the task. Time and Urgency represent the rate at which the output loses value over time; high urgency dictates rapid execution regardless of off-peak deferral opportunities. User Presence denotes the physical or digital availability of the human operator. High presence means the user is actively monitoring the interface and expects a synchronous interaction loop; low presence means the system is operating autonomously and can safely queue or defer work. Model Fit measures the alignment between the task's specific domain (e.g., Python coding, creative writing) and the capability's historical evidence for success in that domain. Output Quality Tier defines the required rigor, accuracy, and depth of the final output, dictating whether a highly capable foundation model or a smaller, faster model is appropriate.Urgency and User Presence represent distinct, orthogonal vectors in the routing algorithm. An urgent task may occur while a user is asleep (e.g., mitigating a critical server outage), requiring high autonomy and rapid execution. Conversely, a user may be actively present for a non-urgent task (e.g., brainstorming long-term strategy), allowing for slower, highly interactive routing choices that do not require immediate resolution.Resolution of Conflicting Routing ScenariosThe interaction of these five factors creates complex routing scenarios. The SAW algorithm mathematically resolves these conflicts by balancing normalized scores against the absolute boundaries of hard restrictions.High Quality Needed but Strict Budget
When a user requests a deep-research compilation but sets a stringent budget limit, the factors naturally conflict. The hard budget cap acts as a pre-scoring filter, immediately eliminating expensive, top-tier commercial models regardless of their quality scores. During scoring, the highest-quality remaining options (e.g., highly capable local open-weight models) win the MCDA evaluation. The Scheduler may subsequently decide to defer the task to a batch-API window to afford a higher-quality commercial model at a discounted rate, provided the delay does not violate the user's urgency constraints.User is Waiting but the Task is Not Urgent
If the user is present at the terminal, but the task itself (e.g., organizing a local photo directory) has no external time sensitivity, the routing algorithm must prioritize the human-computer interaction loop. Because User Presence is high, the router heavily penalizes batch deferrals or long queues. It selects a fast, capable local model to maintain a seamless interaction, even if a cheaper deferral window exists later in the day.Task is Urgent but the User is Unavailable
When a system monitoring script triggers an urgent log analysis while the user is away, User Presence is low, but Urgency is high. The router heavily weights speed and autonomous capability. It will not select routes that have a historical tendency to require mid-task human clarification. However, if the task requires touching sensitive systems, the hard-rule Approval Gate will trigger, overriding the urgency preference to maintain safety. The task will block until the user returns, demonstrating that safety rules supersede urgency vectors.Low-Risk Background Work that Can Be Delayed for Lower Cost
For a request to summarize old meeting notes, Urgency is low, Presence is low, and Cost sensitivity is active. The SAW algorithm heavily rewards the cheapest options. Because the work is low-risk and non-urgent, the Scheduler transitions the task to a "Defer" state, actively queuing it for a scheduled batch-API run or waiting until a local model cluster is idle, perfectly optimizing the cost vector.High-Risk Work that Needs Approval Even When Urgent
If a user requests the orchestration to drop a database table during a live incident, the routing factors will quickly calculate the fastest, most capable route. However, the task facts flag the action as highly irreversible. The hard-rule framework intercepts the normative scoring output. The task is routed to a "Blocked/Ask the User" state, pausing all execution until a cryptographic or manual approval is logged, regardless of how high the urgency factor was weighted.A Cheap Option that Cannot Meet the Required Quality
When a user requests production-grade C++ refactoring and prefers low cost, a cheap, small-parameter local model will be evaluated. While it scores perfectly on the Cost factor, its capability evidence shows high failure rates for complex C++ refactoring. The Model Fit and Quality Tier scores drop to near zero. The weighted aggregate score falls below that of a moderately priced, high-capability model, ensuring OCC does not sacrifice foundational competence for arbitrary savings.Scheduling and Deferral Policy MapThe OCC Scheduler determines the temporal state of a task. This distinction forms the basis of the split-gate architecture.Hard restrictions operate as boolean pass/fail logic gates. Failure to meet any of these criteria results in the immediate and irrevocable removal of the capability from the candidate pool for that specific task. The criteria for hard restrictions encompass physical limitations, access rights, and security policies. Tool Access dictates whether the capability possesses the specific tool, such as a Python interpreter or web scraper, required by the task facts. File Permissions evaluate whether the capability can read or write to the directories specified in the request. Privacy Limits act as a strict barrier; if the task contains personally identifiable information or proprietary code, the capability's evidence must confirm it does not train on user data, or it is eliminated. Context Window Limits ensure that the estimated input size is physically smaller than the capability's maximum context window, preventing mid-execution truncation failures. Budget Caps ensure the estimated cost of the run falls below the user's explicit limit. Reversibility and Safety Limits dictate whether the capability is authorized to perform high-risk actions, such as database deletions. Finally, Availability restricts the pool to capabilities that are currently online and free of active outage alerts.Normal tradeoffs, conversely, are evaluated on a continuous spectrum using the SAW algorithm. If a capability survives all hard restrictions, it is assessed against these weighted criteria. These factors do not eliminate a candidate; they determine its rank. Cost Efficiency rewards lower estimated costs with higher scores. Execution Speed rewards lower estimated latency and higher throughput. Model Fit assigns higher scores to capabilities with strong historical success rates for the specific task domain. Output Quality rewards capabilities that consistently output high-tier results, scaled proportionately by the task's specific quality requirement. The router must be fundamentally transparent. This record acts as the definitive audit trail for why OCC assigned a specific task to a specific capability, moving away from black-box LLM decision-making to a verifiable ledger.The decision record must capture the state of the system at the exact moment of routing. The record begins with Delegation Metadata, capturing a unique cryptographic hash of the routing decision, a reference to the parent task ID, and the exact timestamp of the decision. This is followed by the Ingested Facts, providing a complete snapshot of the task facts at the moment of routing—including the inferred goal, urgency, presence, quality, and risk level—alongside a snapshot of the specific capability metrics, such as prices and health, used during the decision.To explain why certain paths were not taken, the record logs Filtering Outcomes. This includes an array of capabilities rejected during the Pre-Scoring Phase and the specific hard-rule failure for each eliminated candidate (e.g., "Model_A rejected: Insufficient Context Window"). This prevents user confusion regarding why a favorite model was ignored.The core of the explanation lies in the Scoring Mathematics. The record details the selected winning capability, its final SAW aggregate score, the runner-up routes and their respective scores, and the exact user weights applied during the calculation. This provides mathematical proof of the decision.Finally, the record captures the Scheduling State, noting the initial outcome of the scheduling evaluation (e.g., "Run Immediately", "Deferred", "Blocked") and any upstream dependencies blocking execution. A Mutation Log acts as an append-only array tracking any subsequent state changes, such as a task moving from a deferred queue to active execution, or a human approval being granted. This structured approach ensures that the OCC graph UI can accurately draw connections between tasks, visualize the execution pipeline, and allow a user to click on any node to see exactly the rationale behind the orchestration.Uncertainty, Failure, and Fallback BehaviorOrchestration systems inevitably encounter missing information, stale evidence, or mid-execution failures. OCC must handle these conditions gracefully, ensuring system stability without relying on uncontrolled LLM retry loops that compound errors or inflate costs. The system achieves this by borrowing mechanisms from Risk-Based Authentication (RBA) for dynamic verification and the Saga architectural pattern for state compensation.When important task facts are unknown or capability evidence is stale or contradictory, OCC adopts a highly pessimistic stance. For example, if a provider officially claims 99% uptime, but direct OCC telemetry shows 50% failure rates over the last hour, the system defaults to the most restrictive assumption. The router blocks the execution and issues a clarification request to the user. This mechanism operates identically to an RBA "step-up authentication" challenge. Rather than proceeding on a guess, the user is asked to manually input the missing parameter or explicitly override the contradictory evidence before the routing pipeline resumes.If a selected capability fails during active execution—such as an API timing out or a generated script crashing—OCC does not automatically retry the same route. Automated retries on side-effect-producing actions lead to corrupted states. Instead, OCC evaluates the reversibility of the failed action. If the capability altered local state, OCC executes a "compensating transaction" to revert the system to its original state. This is a foundational principle of the Saga pattern used in distributed microservices. Only after the state is successfully compensated does OCC update the Capability Registry to log the failure, degrading the capability's reliability score. The task is then routed back to the scoring phase to select the next best alternative.When a needed approval is not given within a reasonable timeframe, or a user changes priorities mid-execution, OCC must react dynamically. If a task requires human approval but the user times out, the task is safely cancelled and moved to a terminal state. If priorities change (e.g., the user aggressively lowers the budget cap), OCC evaluates the new constraints against the running task. If the task violates the new constraints, it is intercepted and gracefully halted; otherwise, the changes apply only to subsequent routing decisions.In the event that no valid route is available—meaning the Pre-Scoring Eligibility Filter eliminates every single capability in the registry—the router halts immediately. It generates a detailed failure record stating exactly which constraints caused the total elimination. This allows the user to either adjust the task facts (e.g., lowering the context window requirement) or provision new capabilities (e.g., adding an API key for a larger model).Adjacent Systems and Improvement OpportunitiesTo ensure OCC does not operate in a vacuum, its routing architecture borrows proven mechanisms from leading operational dispatch, scheduling, and decision-making systems. take mathematical surveys.Specific Analysis of Scoring versus Hard-Rule-Gate OrderThe active OCC architecture previously listed "Weighted Scoring" before "Hard-Rule Gates." Subsequent research into the Capability Registry (R2) identified that applying restrictions might require removing invalid options prior to scoring. The core research question asks whether this constitutes a true architectural conflict or simply two layers of the same decision process, and whether restrictions should be checked before scoring, after scoring, or both.This analysis confirms that this is not an architectural conflict, but rather a misinterpretation of gating mechanisms. Evaluating all hard-rule gates entirely after scoring is computationally and operationally flawed, as the system would waste resources scoring capabilities that lack the basic permissions to execute the task. Conversely, evaluating them entirely before scoring is incomplete, as certain gates depend on the outcome of the scoring phase.The systems analysis of Kubernetes, Google Borg, and Apache Camel demonstrates that highly efficient routers use a split-gate architecture.Pre-Scoring Filters establish basic eligibility. Static constraints—such as tool access, privacy compliance, file permissions, and input context limits—must be checked before scoring. The primary benefit of this approach is that it eliminates wasted compute and prevents normalization algorithms in the SAW scoring phase from being skewed by a "perfect" but ineligible model. The risk is that if filters are overly aggressive or poorly defined, the candidate pool drops to zero, halting the system prematurely.Post-Scoring Gates enforce operational verification and budget controls. Dynamic constraints—such as verifying the user's total budget cap against the specifically chosen route, or enforcing a human-approval hold based on the winning model's risk profile—must be checked after scoring. The benefit here is that the system ensures the exact cost of the selected route is authorized, and allows the orchestrator to pause at an approval gate holding the specific, intended execution plan for the user to review. The risk is a slightly higher latency before final dispatch, but this is negligible compared to the safety benefits.Therefore, OCC must adopt both Pre-Scoring Filters and Post-Scoring Gates as distinct steps within the routing pipeline.Decision-Ready ConclusionsTo transition the OCC router from theoretical architecture to active implementation, the following conclusions are established based on the evidence gathered.What OCC Needs Before It Can Make Routing Decisions Responsibly
First, MainOrch must officially update the router pipeline to reflect the split-gate architecture: Classifier → Pre-Scoring Filters → Cost Layer → MCDA Scoring (SAW) → Scheduler → Post-Scoring Gates → Decision Logging. Second, OCC requires a normalized attribute engine; a deterministic mathematical function utilizing Simple Additive Weighting must be built to convert the five routing factors into standardized vectors. Finally, OCC needs strict JSON Decision Logs, defining a rigid data schema for routing records to ensure the UI graph can immediately trace all parent/child task links and scoring rationales.What Can Wait Until Later
While the Scheduler supports waiting for dependencies, extracting complex DAG (Directed Acyclic Graph) relationships from unstructured user prompts can wait for advanced classifier iterations. Additionally, the implementation of Semantic Circuit Breakers—the ability to detect logical, contextual hallucinations mid-execution—is deferred. Fallbacks will rely on hard API errors and explicit Saga compensation triggers in the interim.What Remains Unproven
The assumption that local open-weight models can consistently rank high on the "Model Fit" and "Quality Tier" axes remains unproven. This requires live capability telemetry to validate before users can trust the routing algorithm to default to local models. Furthermore, user tolerance for RBA Step-Ups is unknown. It is unclear how much friction users will tolerate when OCC blocks execution to ask for missing task facts before they abandon the interface.What Must Be Tested with Benchmarks or Real Runs
The default baseline weights for the SAW algorithm (Cost, Urgency, Presence, Fit, and Quality) must be empirically tested against a high volume of real-world prompts to ensure the algorithm does not produce counter-intuitive routing assignments. Additionally, the effectiveness of Saga-style compensating transactions must be tested on live file systems to verify that OCC can reliably "undo" a failed agentic action without causing secondary corruption.Specific Decision Required from MainOrch
MainOrch must formally ratify the split-gate architecture (delineating Pre-Scoring Filters and Post-Scoring Gates) and the adoption of Simple Additive Weighting (SAW) as the sole deterministic mechanism for tradeoff evaluation. No further architecture planning regarding router logic should proceed until this modification is committed to the OCC identity documentation.
